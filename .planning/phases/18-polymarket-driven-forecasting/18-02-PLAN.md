<objective>
Extend the API layer to serve Polymarket comparison data alongside forecasts.

Purpose: The frontend needs two things from the API: (1) comparison metadata on each forecast card (badge type, market price, divergence), and (2) a dedicated endpoint for the ComparisonPanel showing all head-to-head comparisons. The ForecastResponse DTO gains an optional `polymarket_comparison` field. Two new calibration endpoints serve the comparison panel data and sparkline snapshots.

Output: Extended ForecastResponse schema, enriched /forecasts/top response, two new calibration endpoints (comparisons list + snapshot time-series), and forecast_service enrichment logic.
</objective>

    **forecast.py schema -- Add PolymarketComparisonData DTO:**

    Add a new Pydantic model at the end of forecast.py (before ForecastResponse):

    ```python
    class PolymarketComparisonData(BaseModel):
        """Polymarket comparison data attached to a forecast card."""
        model_config = ConfigDict(from_attributes=True)

        comparison_id: int = Field(..., description="PolymarketComparison row ID")
        polymarket_event_id: str = Field(..., description="Polymarket event identifier")
        polymarket_title: str = Field(..., description="Polymarket question title")
        polymarket_price: float | None = Field(None, description="Current Polymarket market price (0-1)")
        geopol_probability: float | None = Field(None, description="Latest Geopol probability")
        divergence: float | None = Field(None, description="Geopol probability - Polymarket price")
        provenance: str = Field(..., description="'polymarket_driven' or 'polymarket_tracked'")
        status: str = Field(..., description="'active' or 'resolved'")
        polymarket_slug: str = Field("", description="Polymarket event slug for URL construction")
        geopol_brier: float | None = Field(None, description="Geopol Brier score (resolved only)")
        polymarket_brier: float | None = Field(None, description="Polymarket Brier score (resolved only)")
    ```

    **ForecastResponse -- Add optional polymarket_comparison field:**

    Add to ForecastResponse class (after `expires_at`):
    ```python
    polymarket_comparison: Optional[PolymarketComparisonData] = Field(
        None, description="Polymarket comparison data if this forecast is linked to a market"
    )
    ```

    **forecast_service.py -- Enrich forecasts with comparison data:**

    Add method to ForecastService:
    ```python
    async def enrich_with_comparisons(
        self, forecasts: list[ForecastResponse],
    ) -> list[ForecastResponse]:
        """Attach polymarket_comparison data to forecasts that have linked comparisons."""
    ```

    Implementation:
    1. Collect all forecast_ids from the list
    2. Single query: `SELECT * FROM polymarket_comparisons WHERE geopol_prediction_id IN (:ids)`
    3. Also query Prediction.provenance for each forecast_id to determine badge type
    4. For each matched comparison, build `PolymarketComparisonData` with:
       - `divergence = geopol_probability - polymarket_price` (if both non-null)
       - `provenance` from the Prediction row (polymarket_driven or polymarket_tracked; for tracked, use the fact that the comparison exists but provenance is null on the Prediction -- use "polymarket_tracked" when Prediction.provenance is null but a comparison exists)
    5. Attach to the matching ForecastResponse via `.model_copy(update={"polymarket_comparison": data})`
    6. Return the enriched list

    **forecasts.py route -- Wire enrichment into /forecasts/top:**

    In the `get_top_forecasts` route handler, after building the list of ForecastResponse objects, call:
    ```python
    service = ForecastService(db)
    enriched = await service.enrich_with_comparisons(forecasts)
    ```

    Return the enriched list. This makes comparison data available to every forecast card on every screen (dashboard, globe drill-down) without additional API calls.

    Also wire enrichment into `get_forecasts_by_country` and `get_forecast_by_id` routes for consistency.

    **Important:** The enrichment must be efficient. Use a single batch query with IN clause, not N+1 queries. The comparison table is small (tens of rows), so this is negligible overhead.

    **Add two new endpoints to calibration.py:**

    **Endpoint 1: GET /calibration/polymarket/comparisons**
    - Response model: `ComparisonPanelResponse` (new Pydantic model, inline in calibration.py like existing models)
    - Returns all active + resolved comparisons in a single list, ordered by created_at DESC
    - Each item includes: id, polymarket_event_id, polymarket_slug, polymarket_title, geopol_prediction_id, match_confidence, polymarket_price, geopol_probability, status, geopol_brier, polymarket_brier, polymarket_outcome, resolved_at, created_at, provenance (from joined Prediction row)
    - Uses `PolymarketComparisonService.get_all_comparisons()` (added in Plan 01)
    - Depends on verify_api_key

    Response models (inline):
    ```python
    class ComparisonPanelItem(BaseModel):
        id: int
        polymarket_event_id: str
        polymarket_slug: str
        polymarket_title: str
        geopol_prediction_id: str
        match_confidence: float
        polymarket_price: float | None
        geopol_probability: float | None
        divergence: float | None  # computed: geopol - polymarket
        status: str  # active | resolved
        provenance: str  # polymarket_driven | polymarket_tracked
        geopol_brier: float | None
        polymarket_brier: float | None
        polymarket_outcome: float | None
        resolved_at: str | None
        created_at: str

    class ComparisonPanelResponse(BaseModel):
        comparisons: list[ComparisonPanelItem]
        total: int
    ```

    The route handler builds the service the same way as `get_polymarket_comparisons` (wrapping the db session in a factory), calls `get_all_comparisons()`, and enriches each item with provenance by looking up the joined Prediction row's provenance field. For items where Prediction.provenance is null but a comparison exists, use "polymarket_tracked".

    **Endpoint 2: GET /calibration/polymarket/comparisons/{comparison_id}/snapshots**
    - Path parameter: `comparison_id: int`
    - Query parameter: `limit: int = 30` (max 100)
    - Response model: `SnapshotResponse` (list of snapshot points)
    - Uses `PolymarketComparisonService.get_snapshots_for_comparison(comparison_id, limit)`
    - Returns sampled time-series data for sparkline rendering

    Response model:
    ```python
    class SnapshotPoint(BaseModel):
        polymarket_price: float
        geopol_probability: float
        captured_at: str

    class SnapshotResponse(BaseModel):
        comparison_id: int
        snapshots: list[SnapshotPoint]
        total_available: int  # total snapshots before sampling
    ```

    The endpoint returns the most recent `limit` snapshots. If the total exceeds `limit`, sample evenly (e.g., take every Nth row to get exactly `limit` points spanning the full time range).

    **Important:** Both endpoints require API key auth (Depends(verify_api_key)). Use the same session-wrapping pattern as existing calibration endpoints.
