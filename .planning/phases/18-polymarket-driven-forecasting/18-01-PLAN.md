<objective>
Build the auto-forecast pipeline that transforms Polymarket from a passive comparison tool into an active forecast driver.

Purpose: When Polymarket has a high-volume geopolitical question without a Geopol match, the system automatically generates a competing prediction via the full EnsemblePredictor pipeline. Active comparisons are re-forecasted daily to track probability divergence. This is the core engine that makes Phase 18 work -- everything else (badges, UI panels) depends on this pipeline producing data.

Output: PolymarketAutoForecaster class, DB schema extensions (provenance + polymarket_event_id on Prediction), settings for volume threshold and daily caps, Alembic migration, and wiring into the existing _polymarket_loop.
</objective>

    **auto_forecaster.py (NEW):**

    Create `PolymarketAutoForecaster` class with the following design:

    ```python
    class PolymarketAutoForecaster:
        """Generate Geopol forecasts for unmatched high-volume Polymarket questions."""
    ```

    Constructor takes: `async_session_factory`, `gemini_client: GeminiClient`, `settings: Settings`.

    **Primary method: `async def run(self, geo_events: list[dict], tracked_ids: set[str]) -> dict[str, int]`**

    Flow:
    1. **Filter candidates:** From geo_events, keep those where:
       - `event_id not in tracked_ids` (not already matched by Phase 13 matcher)
       - Volume >= `settings.polymarket_volume_threshold`
       - No existing Prediction row with matching `polymarket_event_id` (dedup query)
       - `compute_horizon_days(event.endDate)` returns a valid horizon (7-365 days)
    2. **Sort by volume DESC**, take up to `settings.polymarket_daily_new_forecast_cap - count_today_new_forecasts()` candidates
    3. **For each candidate:**
       a. Extract country via `extract_country_heuristic(title, tags)` -- if None, fall back to `extract_country_llm(title, description)` via GeminiClient
       b. Extract CAMEO category via GeminiClient (always LLM, use the prompt from RESEARCH.md)
       c. Compute horizon from endDate
       d. Check `gemini_budget_remaining()` -- skip if exhausted
       e. Run `EnsemblePredictor.predict(question=title)` via `asyncio.to_thread`
       f. Persist via `ForecastService.persist_forecast()` with `provenance="polymarket_driven"` and `polymarket_event_id=event_id`
       g. Create `PolymarketComparison` row linking the new prediction to the Polymarket event (match_confidence=1.0 since it's auto-generated, not matched)
       h. Increment Gemini usage counter
    4. Return `{"candidates": N, "generated": M, "skipped_budget": K, "skipped_cap": J}`

    **Helper functions (module-level):**
    - `compute_horizon_days(end_date_str: str | None) -> int | None` -- parse ISO 8601, clamp 7-365 days, return None if outside range or unparseable. Use the exact implementation from RESEARCH.md.
    - `extract_country_heuristic(title: str, tags: list[dict]) -> str | None` -- match against `COUNTRY_NAME_TO_ISO` from advisory_poller.py. Check both title and tag labels (lowercased).
    - `async def extract_country_llm(title: str, description: str, gemini: GeminiClient) -> str | None` -- ask Gemini to infer most relevant country ISO code. Return None on failure.
    - `async def extract_cameo_category(title: str, gemini: GeminiClient) -> str` -- use the CAMEO extraction prompt from RESEARCH.md. Parse JSON response, return cameo_root_code. Default to "14" (protest) on parse failure.
    - `async def count_today_new_forecasts(session: AsyncSession) -> int` -- count predictions with provenance="polymarket_driven" created today (midnight-to-midnight UTC).
    - `async def count_today_reforecasts(session: AsyncSession) -> int` -- count predictions with provenance="polymarket_driven" updated today. Actually -- re-forecasts OVERWRITE existing rows (per RESEARCH.md recommendation), so track via snapshot count or a separate counter. Simplest: count snapshots captured today for active comparisons where geopol_prediction has provenance="polymarket_driven".

    **Secondary method: `async def reforecast_active(self) -> dict[str, int]`**

    Flow:
    1. Query active PolymarketComparison rows where the linked Prediction has `provenance in ("polymarket_driven", "polymarket_tracked")`
    2. Check daily reforecast cap: `settings.polymarket_daily_reforecast_cap - count_today_reforecasts()`
    3. For each comparison (up to cap):
       a. Check `gemini_budget_remaining()` -- skip if exhausted
       b. Run `EnsemblePredictor.predict(question=prediction.question)` via `asyncio.to_thread`
       c. **Overwrite** the existing Prediction row: update `probability`, `scenarios_json`, `ensemble_info_json`, `calibration_json`, `created_at` to now. Do NOT create a new row -- historical values are preserved in `polymarket_snapshots`.
       d. Increment Gemini usage counter
    4. Return `{"active_comparisons": N, "reforecasted": M, "skipped_budget": K}`

    **Anti-patterns to avoid:**
    - Do NOT run EnsemblePredictor synchronously in the event loop. Always `asyncio.to_thread()`.
    - Do NOT create a new GeminiClient per forecast. Reuse the one passed to the constructor.
    - Do NOT create new Prediction rows for re-forecasts. Overwrite existing rows.
    - Do NOT poll Polymarket per-market for prices. Use bulk geo_events data already fetched.

    **comparison.py extension:**
    Add method `async def get_snapshots_for_comparison(self, comparison_id: int, limit: int = 30) -> list[dict]`:
    - Query `PolymarketSnapshot` rows for the given comparison_id, ordered by `captured_at DESC`, limited to `limit`
    - Return list of dicts with keys: `polymarket_price`, `geopol_probability`, `captured_at` (ISO string)
    - If more than `limit` snapshots exist, sample evenly (take every Nth snapshot to get ~`limit` points)
    - This method serves the sparkline data for expanded forecast cards

    Also add method `async def get_all_comparisons(self) -> list[dict[str, Any]]`:
    - Returns both active AND resolved comparisons in a single list, ordered by created_at DESC
    - Each entry includes: id, polymarket_event_id, polymarket_slug, polymarket_title, geopol_prediction_id, match_confidence, polymarket_price, geopol_probability, status, geopol_brier (if resolved), polymarket_brier (if resolved), polymarket_outcome (if resolved), resolved_at (if resolved), created_at
    - This serves the ComparisonPanel in Plan 03


Refactor _polymarket_loop to pass geo_events from the matching cycle to the auto-forecaster (avoid double-fetching from Polymarket API). Extract tracked_ids from the matching cycle result or query PolymarketComparison table.

This avoids coupling the matching cycle's internals. The double-fetch of geo_events is unavoidable unless we restructure, but since `fetch_geopolitical_markets` is already cached by the PolymarketClient session, it's fine.

Add a check: if `datetime.now(UTC).hour == 0` (midnight cycle) or use a module-level flag `_last_reforecast_date` to ensure at-most-once-daily. Simpler: check if any reforecasts have already been done today by counting today's reforecasts -- if count > 0, skip.
