  <action>
    All panels follow the same interface pattern:
    ```typescript
    interface AdminPanel {
      mount(container: HTMLElement): Promise<void>;
      destroy(): void;
    }
    ```
    Each panel takes `AdminClient` in its constructor. Use `h()` from `@/utils/dom-utils` for DOM construction throughout.

    **1. ProcessTable** (`frontend/src/admin/panels/ProcessTable.ts`):
    - `class ProcessTable` implements AdminPanel.
    - `mount()`: Calls `adminClient.getProcesses()`, renders a compact HTML `<table>` with columns: Status (colored dot), Name, Last Run (relative time like "3m ago"), Next Run (relative or "--"), Success/Fail (count/count), Action (trigger button).
    - Dense htop-style: small row height, monospace font for times, no extra padding.
    - Trigger button per row: on click, calls `adminClient.triggerJob(daemon_type)`, immediately updates the row status dot to "running" (green pulse animation), then after 2s re-fetches process list to update.
    - Auto-refresh: `setInterval` every 15s calls `getProcesses()` and re-renders the table body (not the whole table -- preserve DOM structure).
    - Relative time helper: convert ISO datetime to "Xm ago", "Xh ago", "Xd ago".
    - `destroy()`: clear interval, remove element.

    **2. ConfigEditor** (`frontend/src/admin/panels/ConfigEditor.ts`):
    - `class ConfigEditor` implements AdminPanel.
    - `mount()`: Calls `adminClient.getConfig()`, renders a form with grouped settings.
    - Group settings by prefix: "GDELT" (gdelt_*), "RSS" (rss_*), "Gemini" (gemini_*), "Polymarket" (polymarket_*), "ACLED" (acled_*), "Calibration" (calibration_*), "Monitoring" (smtp_*, alert_*, feed_*, drift_*, disk_*), "Other" (remaining).
    - Each group is a collapsible section with header.
    - Each setting renders as:
      - Label: setting key (human-readable, replace underscores with spaces, title case)
      - Description text (from ConfigEntry.description)
      - Input: type=number for int/float, type=text for str, checkbox for bool, textarea for list. Use the `type` field from ConfigEntry.
      - If `editable: false`: show value as read-only text, masked for secrets (show "********").
      - If `dangerous: true`: add a red border-left indicator.
    - Inline validation: on blur/change, validate type constraints (e.g. int fields must be numeric, poll intervals must be > 0). Show red border + error text under invalid fields.
    - Save button: disabled until at least one field has changed AND all fields valid. On save:
      - Collect changed fields only (diff against original values).
      - If any changed field is `dangerous`, show a confirmation dialog: "Changing {field} to {value} may {impact description}. Continue?" with Cancel/Confirm buttons.
      - Call `adminClient.updateConfig(changes)`. On success, show brief "Saved" toast. On error, show error message.
    - "Revert to Defaults" button: calls `adminClient.resetConfig()` which issues `DELETE /api/v1/admin/config`. This deletes all system_config rows, causing all settings to fall back to their Settings (env var / default) values. On success, re-fetches config and re-renders the form. Show confirmation dialog first: "This will reset ALL runtime config overrides to their default values. Continue?"
    - `destroy()`: remove element.

    **3. LogViewer** (`frontend/src/admin/panels/LogViewer.ts`):
    - `class LogViewer` implements AdminPanel.
    - `mount()`: Creates the log viewer UI and starts polling.
    - Header area with:
      - Severity pill toggles: 3 buttons (ERROR, WARN, INFO) with color-coded backgrounds. Click toggles the pill on/off (client-side filter). Default: all ON.
      - Text search input: filters displayed entries by substring match on message field. Debounce 200ms.
      - Active subsystem filter chip: shows current subsystem filter with X to clear. Hidden when no subsystem filter active.
    - Log entries area: a scrollable `<div>` with monospace font, one line per entry.
    - Each entry renders as: `[HH:MM:SS] [SEVERITY] [subsystem] message`.
      - SEVERITY colored: ERROR=#ef4444, WARN=#f59e0b, INFO=#888.
      - Subsystem is clickable -- clicking it sets the subsystem filter.
    - Auto-scroll behavior:
      - By default, container is scrolled to bottom (latest entries visible).
      - On manual scroll up (scrollTop < scrollHeight - clientHeight - 50), set `autoScroll = false`. Show a "Resume auto-scroll" button at the bottom.
      - Clicking resume button scrolls to bottom and re-enables autoScroll.
      - When autoScroll is true and new entries arrive, scroll to bottom.
    - Polling: `setInterval` every 3s calls `adminClient.getLogs({ severity, subsystem })`.
      - Server-side filters severity and subsystem to reduce payload.
      - Client-side additionally filters by search text and pill toggles.
      - Diff-based update: compare new entries against displayed entries (by timestamp+message). Only append truly new entries -- don't re-render the entire list.
    - `destroy()`: clear interval, remove element.

    **4. SourceManager** (`frontend/src/admin/panels/SourceManager.ts`):
    - `class SourceManager` implements AdminPanel.
    - `mount()`: Calls `adminClient.getSources()`, renders a card grid.
    - Card grid: 2 or 3 columns (responsive via CSS grid, `grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))`).
    - Each source renders as a card:
      - Header: source name + tier badge (if RSS: "Tier 1" or "Tier 2" pill).
      - Health indicator: green dot if healthy, red dot if unhealthy, gray if unknown.
      - Last run: relative time.
      - Events count: total events from this source.
      - Enable/Disable toggle switch: styled checkbox. On toggle, calls `adminClient.toggleSource(name, enabled)`. Optimistic UI: toggle immediately, revert on error.
    - Auto-refresh: `setInterval` every 30s re-fetches sources and updates card contents (not full re-render -- update text nodes).
    - `destroy()`: clear interval, remove element.

    **5. Wire panels into admin-layout** (`frontend/src/admin/admin-layout.ts`):
    Update the placeholder section rendering from Plan 02 to instantiate real panels:
    - Import all 4 panel classes.
    - On section navigate callback from sidebar:
      1. If current panel exists, call `destroy()` on it.
      2. Clear content area.
      3. Instantiate the new panel class with the AdminClient.
      4. Call `panel.mount(contentArea)`.
    - Default to ProcessTable on initial load.

    **6. Add panel CSS** (`frontend/src/admin/admin-styles.css`):
    Add styles for all panels, scoped under `.admin-layout`:
    - Process table: `.process-table` with compact rows, `.status-dot` (8px colored circles), `.trigger-btn` (small red-accent button), running pulse animation.
    - Config editor: `.config-group` collapsible sections, `.config-field` with label+input+description+error layout, `.config-field.invalid` red border, `.config-field.dangerous` red-left-border, `.save-btn` and `.revert-btn`, `.confirm-dialog` modal overlay.
    - Log viewer: `.log-viewer` layout, `.severity-pill` toggle buttons with active/inactive states, `.log-entry` monospace rows, `.log-entry .subsystem` clickable with hover underline, `.resume-btn` floating button, `.search-input`.
    - Source manager: `.source-grid` CSS grid, `.source-card` with border, header, health dot, toggle switch styling.
    - Toast notification: `.admin-toast` fixed bottom-right, fade-in/out animation, 3s duration.
