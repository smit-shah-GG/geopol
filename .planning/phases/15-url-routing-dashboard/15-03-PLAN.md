    **1a. Add submission types to api.ts:**
    Add these interfaces matching the backend schemas from Phase 14 (`src/api/schemas/submission.py`):

    ```typescript
    /** POST /forecasts/submit response -- LLM-parsed question */
    export interface ParsedQuestionResponse {
      request_id: string;
      question: string;
      country_iso_list: string[];
      horizon_days: number;
      category: string;
      status: string;
      parsed_at: string;
    }

    /** GET /forecasts/requests item -- submission status */
    export interface ForecastRequestStatus {
      request_id: string;
      question: string;
      country_iso_list: string[];
      horizon_days: number;
      category: string;
      status: 'pending' | 'confirmed' | 'processing' | 'complete' | 'failed';
      submitted_at: string;
      completed_at: string | null;
      prediction_ids: string[];
      error_message: string | null;
    }

    /** POST /forecasts/submit/{id}/confirm response */
    export interface ConfirmSubmissionResponse {
      request_id: string;
      status: string;
      message: string;
    }
    ```

    **1b. Add submission methods to forecast-client.ts:**
    Add these methods to `ForecastServiceClient`:

    `submitQuestion(question: string): Promise<ParsedQuestionResponse>`:
    - POST to `/forecasts/submit` with JSON body `{ question }`.
    - NOT deduplicated, NOT circuit-broken (mutation, user expects immediate feedback).
    - Uses existing `fetchJson` method.

    `confirmSubmission(requestId: string): Promise<ConfirmSubmissionResponse>`:
    - POST to `/forecasts/submit/${encodeURIComponent(requestId)}/confirm`.
    - NOT deduplicated, NOT circuit-broken.

    `getRequests(statusFilter?: string): Promise<ForecastRequestStatus[]>`:
    - GET `/forecasts/requests` with optional `?status_filter=` query param.
    - Deduplicated (polled periodically, safe to dedup).
    - Uses health breaker (low-priority endpoint, shares health circuit).

    Import the new types. Add `ForecastRequestStatus[]` fallback `EMPTY_REQUESTS: ForecastRequestStatus[] = []` for the circuit breaker.

    **2a. Create MyForecastsPanel:**
    Create `frontend/src/components/MyForecastsPanel.ts`:
    - Extends `Panel` base class (import from `./Panel`).
    - Constructor: `super({ id: 'my-forecasts', title: 'MY FORECASTS', showCount: true })`.

    **Refresh behavior:**
    - `refresh()`: calls `forecastClient.getRequests()`, renders the list. Called by RefreshScheduler.
    - `update(requests: ForecastRequestStatus[])`: accepts pre-fetched data (for coordinated loads).

    **Rendering:**
    - Each request renders as a row/card with:
      - Question text (truncated to ~80 chars).
      - Status badge: colored span with text. CSS classes: `status-pending` (yellow/warning), `status-confirmed` (blue/accent), `status-processing` (blue/accent, pulsing), `status-complete` (green/success), `status-failed` (red/critical).
      - Country ISO codes (from `country_iso_list`, joined with comma).
      - Submitted time (relative, e.g., "2h ago").
    - Completed forecasts: clicking a completed request dispatches `forecast-selected` CustomEvent with the first `prediction_id` (load the forecast and open ScenarioExplorer). Use `forecastClient.getForecastById(predictionIds[0])` to fetch the full forecast, then dispatch the event.
    - Failed forecasts: show error message in muted red text below the question.
    - Empty state: "No submitted forecasts yet" message.
    - Sort: most recent first (by `submitted_at`).

    **2b. Create SourcesPanel:**
    Create `frontend/src/components/SourcesPanel.ts`:
    - Extends `Panel` base class.
    - Constructor: `super({ id: 'sources', title: 'DATA SOURCES', showCount: false })`.

    **Data source:** Derives from `HealthResponse.subsystems` (already fetched by SystemHealthPanel's refresh cycle). The subsystems array includes entries for GDELT, RSS, Polymarket, etc. Each has `name`, `healthy`, `detail`, `checked_at`.

    **Rendering:**
    - `update(health: HealthResponse)`: accepts HealthResponse data.
    - Filters subsystems to show data-source-relevant ones: look for subsystem names containing "gdelt", "rss", "polymarket", "chromadb" (case-insensitive).
    - Each source renders as a row:
      - Status dot: green (healthy) / red (unhealthy). Same `.status-dot` CSS class as SystemHealthPanel.
      - Source name (e.g., "GDELT", "RSS Feeds", "Polymarket").
      - Staleness indicator: compute from `checked_at` timestamp. Show relative time ("3m ago", "2h ago"). Color-code: green if < 30min, yellow if 30min-2h, red if > 2h.
      - Detail text from subsystem (e.g., "15,234 events indexed").
    - If no matching subsystems found: show "No source data available -- check /api/v1/health".

    - In `unmountDashboard()`: destroy both panels.
    - On initial data load: fetch requests and pass to MyForecastsPanel. Pass health data to SourcesPanel.
