<objective>
Wire the backtesting engine into the API layer and scheduler infrastructure. Create admin API endpoints for triggering, cancelling, listing, and exporting backtest runs. Integrate with ProcessPoolExecutor via the existing heavy job pattern. Add checkpoint discovery for TiRGN vs RE-GCN model comparison.

Purpose: The backtesting engine from Plan 01 has no way to be triggered or queried. This plan exposes it through the admin API and wires it into the scheduler's heavy job infrastructure so it runs in a subprocess worker with mutual exclusion against other heavy jobs (daily pipeline, polymarket cycle, TKG retrain). Model comparison (BTEST-02) is enabled by checkpoint discovery and the multi-checkpoint run config.

Output: 5 modified files -- admin API endpoints, Pydantic schemas, admin service methods, heavy_runner.py function, job_wrappers.py async wrapper.
</objective>

    Add an `async def heavy_backtest(config_json: str) -> None` function following the exact pattern of `heavy_polymarket_cycle()`:

    1. Import `from src.scheduler.heavy_runner import run_backtest`
    2. Acquire `_heavy_job_lock` (prevents concurrent heavy work)
    3. Get the running loop: `loop = asyncio.get_running_loop()`
    4. Dispatch: `returncode = await loop.run_in_executor(_get_process_executor(), run_backtest, config_json)`
    5. Raise RuntimeError if returncode != 0.
    6. Wrap in try/except, log exception, re-raise.

    CRITICAL: The config_json parameter is a plain string -- it passes through ProcessPoolExecutor pickling without issue. Do NOT pass BacktestRunConfig objects or async session factories.

    IMPORTANT: heavy_backtest() is NOT registered as an APScheduler interval job (locked decision: on-demand only). It's called via asyncio.create_task() from AdminService.start_backtest_run(). This is different from the other heavy wrappers which are APScheduler-triggered.
